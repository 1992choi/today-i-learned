# System Design
- Inflearn > 장애를 허용하는 견고한 시스템 만들기

### 서비스 내부에서 애플리케이션을 나누는 이유
- 장점
  - 장애 격리
    - 하나의 서비스 장애가 전체 시스템으로 확산되지 않고, 영향 범위를 최소화할 수 있음
  - 독립적인 배포와 스케일링
    - 서비스별로 트래픽에 맞춰 개별 확장 및 배포가 가능해 효율적인 운영이 가능함
  - 각각의 애플리케이션을 서로 다른 팀에서 개발 가능
    - 팀 간 의존도를 낮추고 병렬 개발이 가능해 개발 속도를 높일 수 있음
  - 기술 스택, 언어 선택이 자유로움
    - 서비스 특성에 맞는 최적의 기술을 선택하여 성능과 생산성을 동시에 확보할 수 있음
- 단점
  - 복잡성, 운영비용 증가
    - 서비스 간 통신, 배포, 모니터링 등 관리 포인트가 늘어나 운영 난이도가 상승함
  - 데이터 일관성 문제
    - 분산된 데이터 관리로 인해 트랜잭션 처리와 일관성 유지가 어려워질 수 있음

### 이중화와 복제(Replication)
- 이중화와 복제(Replication)
  - 이중화와 복제는 견고한 시스템 설계의 기초이다.
    - 이중화
      - 동일한 역할을 수행하는 시스템을 여러 개 구성하여 하나에 장애가 발생해도 다른 시스템이 즉시 대체할 수 있도록 하는 방식
    - 복제
      - 데이터를 여러 시스템에 동일하게 복사해 저장하고, 장애 시 데이터 유실을 방지하거나 읽기 부하를 분산시키는 방식
- 이중화의 목적
  - 동일한 기능을 하는 시스템 구성 요소를 2개 이상 두어 단일 장애점(SPOF)를 제거하고, 일부 구성요소의 장애시에도 서비스가 지속될 수 있도록 만드는 방법
  - 장애 발생 시에도 트래픽을 정상 인스턴스로 우회하여 서비스 중단을 최소화하는 것이 핵심
- 실무에서의 이중화와 복제
  - 이중화 예시
    - 로드밸런서 + 애플리케이션
      - 여러 애플리케이션 인스턴스를 두고 로드밸런서가 트래픽을 분산하며, 특정 인스턴스 장애 시 자동으로 제외하여 가용성을 확보함
  - 복제 예시
    - DB
      - DB는 상태를 가지기 때문에 단순 이중화가 어렵고, 데이터 일관성을 유지하기 위해 복제 방식을 사용함
      - Primary(Write)는 쓰기를 담당하고, Replica(Read)는 복제된 데이터를 기반으로 조회를 처리하여 부하를 분산함
      - 복제 방식에 따라 Primary의 변경 사항을 Replica가 즉시 반영하지 않을 수 있으며, 동기 복제는 일관성을 보장하지만 지연이 발생하고 비동기 복제는 성능은 좋지만 데이터 불일치 가능성이 존재하는 트레이드오프가 있음

### 확장성 - 성능과 안정성의 연관관계
- 설계에서의 확장성
  - 확장의 종류
    - 수직 확장 (Scale Up)
      - 서버의 CPU, 메모리 등의 자원을 증설하여 성능을 향상시키는 방식으로, 구현은 간단하지만 물리적 한계와 비용 증가가 존재함
    - 수평 확장 (Scale Out)
      - 서버 대수를 늘려 부하를 분산하는 방식으로, 높은 확장성을 제공하지만 분산 처리에 따른 복잡성이 증가함
  - 설계에서의 확장성 정의
    - 리소스를 추가하면 처리량이 올라갈 수 있는 성질
      - 한계
        - DB의 경우 수평 확장에 큰 제약이 있다. (위에 복제 예시에서 언급)
        - 처리량을 올리기 위해 애플리케이션을 수평 확장한다고 해도, DB는 수평 확장이 용이하지 않기 때문에 병목 현상이 발생할 수 있다.
        - 이를 위해서는 다음과 같은 방법을 고려할 수 있다.
          - 샤딩
            - 데이터를 여러 DB로 분산 저장하여 쓰기 및 읽기 부하를 나누는 방식
          - 쿼리 최적화 및 인덱스 추가
            - 불필요한 데이터 접근을 줄이고 조회 성능을 개선하여 DB 부하를 감소시킴
          - 캐싱 활용
            - 자주 조회되는 데이터를 메모리에 저장하여 DB 접근 횟수를 줄임
          - 읽기 전용 복제본 (Read Replica)
            - 읽기 요청을 Replica로 분산시켜 Primary DB의 부하를 완화함
          - 데이터 구조 개선
            - 비정규화 등
              - 조인 비용을 줄이고 조회 성능을 높이기 위해 데이터 중복을 허용하는 설계 방식
          - 트랜잭션 쪼개기
            - 하나의 큰 트랜잭션을 여러 개로 나누어 DB 락과 처리 시간을 줄여 전체 처리량을 개선함

### Retry와 AOP
- Spring Retry 사용
  - Spring Retry 란?
    - 일시적인 장애(네트워크 오류, DB 락 등)에 대해 일정 횟수만큼 자동으로 재시도를 수행하도록 지원하는 라이브러리
  - 사용절차
    1. Spring Retry 의존성 추가 (pom.xml)
    2. 메인 클래스에 @EnableRetry 추가
    3. 재시도할 예외를 상속관계로 구분
    4. @Retryable로 재시도 횟수, 간격 지정
    5. (필요시) 재시도에 대한 Listener 지정
    6. 모든 재시도가 실패했을 때에 대한 처리 (Exception Handling)
  - 어노테이션 예시
    - @Retryable
      - value / include: 재시도를 수행할 예외 클래스 지정
      - exclude: 재시도를 제외할 예외 지정
      - maxAttempts: 최대 재시도 횟수 (기본 3회)
      - backoff: 재시도 간 대기 시간 설정 (delay, multiplier 등으로 점진적 증가 가능)
      - recover: @Recover 메서드와 연계하여 최종 실패 시 대체 로직 수행 가능
- AOP
  - @Retryable와 AOP에 대한 연관관계
    - @Retryable은 AOP 기반으로 동작하며, 프록시를 통해 메서드 호출을 가로채 재시도 로직을 삽입함
    - 즉, 실제 비즈니스 로직을 수정하지 않고도 횡단 관심사(재시도 처리)를 분리하여 적용할 수 있음
  - 주의사항
    - Self Invocation
      - 동일 클래스 내부에서 메서드를 직접 호출하면 프록시를 거치지 않아 @Retryable이 동작하지 않는 문제가 발생함
      - 이를 해결하기 위해서는 별도의 빈으로 분리하거나, 프록시를 통해 호출되도록 구조를 변경해야 함

### Timeout과 장애 전파 방지
- 적절한 타임아웃 설정으로 장애 전파 방지
  - 외부 시스템이나 내부 서비스 호출 시 무한 대기를 방지하고, 일정 시간 내 응답이 없으면 빠르게 실패 처리하여 장애 확산을 차단함
    - - 빠르게 실패 처리하면 해당 요청을 오래 붙잡지 않고 자원을 즉시 반환하여, 스레드 고갈이나 커넥션 풀 고갈을 방지하고 다른 정상 요청이 처리될 수 있도록 함
  - 타임아웃이 없거나 너무 길면 장애가 연쇄적으로 전파되어 전체 시스템 지연 및 장애로 이어질 수 있음
- Timeout 비교
  - Connection Timeout
    - 서버와 연결을 맺는 시간을 제한
    - 네트워크 지연이나 서버 다운 상황에서 빠르게 실패를 감지할 수 있어 불필요한 대기 시간을 줄임
  - Read Timeout
    - 연결된 서버로부터 응답을 받는 시간을 제한
    - 서버가 응답을 늦게 주거나 처리 지연이 발생할 경우 무한 대기를 방지하고, 장애 상황을 빠르게 인지할 수 있음

### Circuit Breaker
- Circuit Breaker 란?
  - 연속적인 장애가 발생하는 외부 시스템 호출을 차단하여, 시스템 전체로 장애가 확산되는 것을 방지하는 패턴
- 처리방식
  - CLOSE
    - 초기 상태로 모든 요청을 정상적으로 외부 시스템에 전달하며, 최근 호출 결과를 기준으로 실패율을 슬라이딩 윈도우 기반으로 집계함
    - 일정 횟수 이상의 호출이 누적된 이후, 설정된 실패율 임계치를 초과하면 OPEN 상태로 전환됨
  - OPEN
    - 외부 시스템 호출을 완전히 차단하고, 즉시 예외를 발생시키거나 fallback 로직을 수행하여 불필요한 리소스 소모를 방지함
    - 이 상태에서는 실제 호출이 발생하지 않기 때문에 장애가 있는 시스템에 부하를 더 이상 주지 않음
  - HALF_OPEN
    - 설정된 대기 시간이 지나면 일부 제한된 요청만 외부 시스템으로 전달하여 정상 동작 여부를 검증함
    - 이때 허용된 요청 수(permitted calls) 내에서 성공 비율을 기준으로 상태를 판단함
    - 일정 비율 이상 성공하면 CLOSE로 복구하고, 실패가 발생하면 즉시 OPEN으로 되돌아감
  - 동작 순서
    - CLOSE 상태에서 모든 요청을 처리하면서 실패율을 지속적으로 집계
    - 최소 호출 수를 만족한 이후, 실패율이 임계치를 초과하면 OPEN 상태로 전환
    - OPEN 상태에서는 외부 호출을 차단하고 설정된 시간 동안 대기
    - 대기 시간이 지나면 HALF_OPEN 상태로 전환하여 일부 요청만 테스트 수행
    - 테스트 요청이 일정 기준 이상 성공하면 CLOSE로 복구, 실패 시 다시 OPEN으로 전환
- 설정
  - 옵션
    - failureRateThreshold
      - 실패율 기준 (예: 50% 이상이면 OPEN 전환)
    - slidingWindowType
      - 실패율을 계산하는 기준 방식으로 COUNT_BASED(요청 개수 기반) 또는 TIME_BASED(시간 기반)으로 설정 가능
    - slidingWindowSize
      - 실패율을 계산하기 위한 요청 수 또는 시간 범위
    - minimumNumberOfCalls
      - Circuit Breaker가 동작하기 위한 최소 호출 수 (이 값 미만일 경우 실패율을 계산하지 않음)
    - waitDurationInOpenState
      - OPEN 상태에서 HALF_OPEN으로 전환되기까지 대기 시간
    - permittedNumberOfCallsInHalfOpenState
      - HALF_OPEN 상태에서 허용할 요청 수
    - recordExceptions
      - 어떤 예외를 실패로 간주할지 정의하며, 지정된 예외에 대해서만 실패율 계산에 포함됨
  - 예시
    - ```
      @CircuitBreaker(
        name = "externalService", // yml에 명시된 key (= resilience4j.circuitbreaker.instances.externalService)
        fallbackMethod = "fallback"
      )
      public String call() {
        return externalApi.call();
      }

      public String fallback(Throwable t) {
        // 장애 발생 시 대체 로직 수행 (로그, 기본값 반환 등)
        return "fallback response";
      }
      ```
    - ```
      // application.yml 예시 (Resilience4j)
      resilience4j:
        circuitbreaker:
          instances:
            externalService:
              failureRateThreshold: 50
              slidingWindowType: COUNT_BASED
              slidingWindowSize: 10
              minimumNumberOfCalls: 5
              waitDurationInOpenState: 10s
              permittedNumberOfCallsInHalfOpenState: 3
              recordExceptions:
                - java.io.IOException
                - java.util.concurrent.TimeoutException
      ```
- 분산 환경에서 Circuit Breaker의 한계
  - 애플리케이션 단위로 동작
    - 여러 인스턴스 중 일부에서만 Circuit Breaker가 OPEN 되고, 다른 인스턴스는 계속 요청을 보내는 문제가 발생함
  - 문제점
    - 목적달성 실패
      - 일부 인스턴스는 계속 외부 시스템에 요청을 보내 장애 차단 효과가 떨어짐
    - 유저 경험 저하
      - 어떤 사용자는 정상 응답을 받고, 어떤 사용자는 실패를 경험하는 불일치가 발생함
- 해결 방법
  - 외부 저장소 활용 (Redis 등)
    - Circuit Breaker 상태를 중앙에서 공유하여 모든 인스턴스가 동일한 상태를 기준으로 동작하도록 만듦
  - Service Mesh 활용
    - 네트워크 레벨에서 Circuit Breaker를 적용하여 애플리케이션과 무관하게 일관된 제어 가능
  - Kafka 활용
    - 상태 변경 이벤트를 발행/구독하여 각 인스턴스가 Circuit Breaker 상태를 동기화함

### Fallback과 Graceful Degradation
- Fallback
  - 주 기능이 실패했을 때 대체 동작을 제공하여 사용자 경험의 완전한 중단을 방지
    - 단순 실패 반환
      - 빠르게 실패 응답을 반환하여 리소스 낭비를 줄이고 장애 확산을 방지함
    - 재시도 큐에 저장
      - 즉시 처리하지 못한 요청을 큐에 적재하고, 이후 재처리를 통해 일관성을 보완함
    - 대체 동작 수행
      - 캐시 데이터 반환, 기본값 제공 등으로 최소한의 기능을 유지함
    - 예외 그대로 던지기
      - 복구가 불가능한 경우 명확하게 실패를 전달하여 상위 레이어에서 처리하도록 함
- Graceful Degradation
  - 시스템의 일부 기능이 실패하더라도 핵심 기능은 유지하여 서비스의 완전한 중단없이 부분적으로 작동
    - 특정 부가 기능이 실패하더라도 핵심 비즈니스 흐름은 유지하는 것이 목적임
    - 예시
      - 상품 판매 페이지에서 별점, 추천 등 부가 기능이 실패하더라도 상품 구매 기능은 정상적으로 제공함

### Rate Limit, Bulkhead
- Rate Limit
  - 요청 속도 제한 (Throughput Control)
    - 단위 시간당 허용되는 요청 수를 제한하여 시스템에 과도한 트래픽이 유입되는 것을 사전에 방지함
    - 토큰 버킷, 리키 버킷 등의 알고리즘을 사용하여 일정량의 요청만 처리하고 나머지는 거부하거나 지연시킴
    - 주로 API Gateway나 인증 서버 앞단에서 적용되어 전체 시스템 보호 역할을 수행함
- Bulkhead
  - 자원 격리 (Isolation)
    - 시스템 자원을 여러 영역으로 분리하여 특정 영역의 장애나 과부하가 전체 시스템으로 확산되는 것을 방지함
    - 예를 들어, 특정 API에 대해 별도의 스레드 풀을 할당하여 다른 API의 처리에 영향을 주지 않도록 구성함
    - 하나의 리소스가 고갈되더라도 다른 기능은 정상적으로 동작할 수 있도록 설계하는 것이 핵심
- Circuit Breaker / Rate Limiter / Bulkhead 비교
  - Circuit Breaker
    - 높은 실패율이나 장애 상황에서 외부 호출을 차단하여 추가적인 부하와 장애 확산을 방지함 (사후 대응)
  - Rate Limiter
    - 높은 부하를 예방하기 위해 단위 시간당 처리 가능한 요청 수를 제한함 (사전 제어)
  - Bulkhead
    - 높은 부하를 예방하기 위해 동시에 처리할 수 있는 스레드 수를 제한하고 자원을 분리하여 영향 범위를 격리함 (격리 전략)

### 장애 (거의) 없는 서비스로 가는 길
- 장애 없는 서비스로 가기 위해 필요한 것들
  - 전체 시스템의 가시성 확보 (관측 가능성의 중요성)
    - 시스템을 관측할 수 있는 상태가 아니면, 트래픽이 얼마나 들어오는지 자체를 알 수 없다.
    - 로그나 Latency에 대한 정보없이는 간헐적으로 시간이 오래 걸린다는걸 알기 어렵다.
    - 이런 정보를 파악하고자 아래와 같은 기술을 도입할 수 있다.
      - Logs
        - 애플리케이션의 이벤트와 에러 상황을 기록하여 문제 발생 원인을 추적할 수 있음
      - Metrics
        - CPU, 메모리, 요청 수, 응답 시간 등 수치 데이터를 기반으로 시스템 상태를 실시간으로 파악함
      - Traces
        - 하나의 요청이 여러 서비스들을 거치는 흐름을 추적하여 병목 지점과 지연 원인을 분석함
  - 시스템에 대한 테스트
    - 비즈니스적으로 의미있는 테스트 코드들을 작성하고, 유지하는게 중요
    - 필요하다면 카오스 엔지니어링도 시도해 볼 수 있음
      - 카오스 엔지니어링이란?
        - 의도적으로 장애를 주입하여 시스템이 실제 장애 상황에서도 정상적으로 동작하는지 검증하는 기법
        - 네트워크 지연, 서버 다운, 트래픽 증가 등을 인위적으로 발생시켜 복원력과 대응 능력을 테스트함
  - 배포 전략과 자동화
    - 테스트로 모든 장애를 예방할 수 없음. 따라서 새로운 코드를 배포할 때는
      - 영향 받는 사용자 숫자가 적도록 점진적으로 배포
        - 카나리 배포
          - 전체 사용자 중 일부에게만 신규 버전을 먼저 적용하고, 문제가 없을 경우 점진적으로 확산하는 방식
        - 블루그린 배포
          - 기존 버전(Blue)과 신규 버전(Green)을 동시에 운영하다가, 트래픽을 한 번에 전환하여 배포하는 방식
      - 빠르고 자동으로 롤백할 수 있는 환경 구성
        - 배포 실패 시 이전 버전으로 즉시 되돌릴 수 있도록 자동화된 롤백 전략을 마련해야 함
      - 기능 플래그 배포처럼 배포는 미리 해두고 간단히 설정값 등을 변경하여 적용
        - 코드 배포와 기능 활성화를 분리하여 위험도를 낮추고 빠른 대응이 가능함
  - 아키텍처 수준에서의 고민
    - 비동기 통신과 느슨한 결합
      - 서비스 간 의존도를 낮추고, 일부 장애가 전체 시스템으로 전파되지 않도록 구조를 설계함
  - 장애 대응 및 회고
    - 장애는 완전히 막을 수 없음
    - 그러므로 장애 발생 시 빠르게 대응할 수 있도록 장애 대응 프로세스를 마련해야함
      - 장애 감지와 알림, On-call 등
        - 실시간 모니터링과 알림 체계를 통해 장애를 빠르게 인지하고 담당자가 즉시 대응할 수 있도록 구성함
    - 장애 발생 후에는 회고를 통해 '누가 잘못했는가'가 아니라 '무엇이, 왜' 잘못되었는지 시스템과 프로세스 관점에서 분석하고 개선점을 도출

### Kafka 기본 개념과 비동기 통신
- 브로커
  - Kafka 서버 인스턴스로, 메시지를 저장하고 클라이언트(프로듀서/컨슈머)의 요청을 처리하는 역할을 수행함
- 클러스터
  - 여러 브로커로 구성된 집합으로, 데이터 분산 저장과 장애 대응(고가용성)을 위해 구성됨
- 프로듀서
  - 메시지를 생성하여 특정 토픽으로 전송하는 역할을 하며, 어떤 파티션에 보낼지 결정할 수 있음
- 컨슈머
  - 토픽에 저장된 메시지를 읽어 처리하는 역할을 수행함
  - 컨슈머 그룹
    - 여러 컨슈머가 하나의 그룹으로 묶여 파티션을 나눠 처리하며, 각 파티션은 하나의 컨슈머에게만 할당되어 병렬 처리와 중복 소비 방지를 보장함
- 토픽
  - 메시지가 저장되는 논리적인 채널로, 동일한 성격의 데이터가 모이는 단위임
- 파티션
  - 토픽을 여러 개로 나눈 물리적 단위로, 메시지를 분산 저장하여 병렬 처리와 확장성을 제공함
  - 파티션 키
    - 메시지를 특정 파티션에 할당하기 위한 기준 값으로, 동일한 키는 동일한 파티션으로 보내져 순서 보장을 가능하게 함
- 오프셋
  - 파티션 내에서 메시지의 위치를 나타내는 고유 번호로, 컨슈머가 어디까지 읽었는지를 추적하는 기준이 됨

### 메시지 유실과 전송 보장 수준
- At-most-once
  - 메시지가 0번 또는 1번 전달된다. (유실될 수 있음)
    - 메시지를 전달하기 전에 오프셋을 먼저 커밋하는 방식으로, 처리 실패 시 재처리가 되지 않아 유실 가능성이 존재함
    - 지연 없이 빠른 처리가 가능하지만, 데이터 손실을 감수해야 하는 경우에 적합함
- At-least-once
  - 메시지가 최소 1번 이상 전달된다. (중복될 수 있음)
    - 메시지 처리 이후에 오프셋을 커밋하여, 처리 중 장애 발생 시 동일 메시지가 재전송될 수 있음
    - 데이터 유실은 방지할 수 있지만, 중복 처리를 고려한 로직(멱등성)이 필요함
- Exactly-once
  - 메시지가 정확히 1번 전달된다. (유실/중복 없음)
    - 트랜잭션과 idempotent producer 등을 활용하여 중복 없이 한 번만 처리되도록 보장함
    - 구현 난이도가 높고 성능 오버헤드가 존재하지만, 데이터 정합성이 중요한 경우에 사용됨

### DLQ(Dead Letter Queue)와 순서 보장
- DLQ란?
  - 정상적으로 처리되지 못한 메시지를 별도의 큐(토픽)로 분리하여 저장하는 방식
    - 재시도 횟수 초과, 데이터 포맷 오류, 비즈니스 로직 예외 등의 경우 원본 스트림에서 분리하여 장애 전파를 방지함
    - 이후 운영자가 분석하거나, 별도 프로세스를 통해 재처리할 수 있도록 하기 위한 목적
- DLQ를 사용했을 때 순서 보장에 대해
  - 어떤 경우에 순서가 틀어질 수 있는지
    - 예시
      - 파티션 내 메시지 순서: A → B → C
      - A 처리 실패 → DLQ로 이동
      - B, C는 정상 처리 진행
      - 이후 DLQ에 있던 A를 재처리하면 처리 순서가 B, C 이후가 되어 순서가 깨짐
  - 순서 보장이 중요한 경우 보완 방법
    - 실패 메시지를 DLQ로 바로 보내지 않고, 동일 파티션에서 재시도를 블로킹 방식으로 수행하여 순서를 유지함
    - 파티션 키를 잘 설계하여 순서 보장이 필요한 데이터만 동일 파티션으로 묶고, 나머지는 분산 처리함
    - DLQ 재처리 시에도 원래 순서를 고려한 별도 큐나 재처리 로직을 구성함 (예: 순차 재처리)
    - 메시지에 버전, 타임스탬프 등을 포함시켜 소비 단계에서 순서를 보정하는 방식도 고려할 수 있음

### 백엔드 개발자가 알아야할 카프카 주요 설정
- Producer
  - enable.idempotence
    - 프로듀서가 메시지를 재전송하더라도 중복 없이 정확히 한 번만 브로커에 기록되도록 보장하는 설정
    - true로 설정 시 acks=all, retries>0 등의 조건이 함께 요구되며, 데이터 정합성을 높이지만 약간의 성능 오버헤드가 발생함
  - retries
    - 메시지 전송 실패 시 재시도 횟수를 지정
    - 0이면 재시도 없이 즉시 실패 처리, 값이 클수록 일시적인 장애에 강하지만 중복 전송 가능성이 증가 (idempotence로 보완 가능)
  - delivery.timeout.ms
    - 메시지 전송이 완료되기까지 허용되는 전체 시간 (재시도 포함)
    - 이 시간 내에 성공하지 못하면 실패 처리되며, linger.ms + request.timeout.ms + retries와 함께 고려해야 함
  - linger.ms
    - 메시지를 즉시 전송하지 않고 일정 시간 동안 모아서 배치로 전송하기 위해 대기하는 시간
    - 0이면 즉시 전송 (지연 최소화), 값이 클수록 처리량은 증가하지만 응답 지연이 늘어남
  - batch.size
    - 하나의 배치로 묶어 전송할 최대 바이트 크기
    - 클수록 네트워크 효율은 좋아지지만 메모리 사용량 증가 및 지연 가능성 존재
  - compression.type
    - 메시지 압축 방식 설정 (none, gzip, snappy, lz4, zstd 등)
    - 압축률이 높을수록 네트워크 사용량 감소하지만 CPU 사용량 증가 (예: gzip은 압축률 높고 느림, snappy는 빠르지만 압축률 낮음)
- Consumer
  - enable.auto.commit
    - 컨슈머가 주기적으로 자동으로 오프셋을 커밋할지 여부
    - true: 편리하지만 메시지 처리 전에 커밋될 수 있어 유실 가능성 존재
    - false: 수동 커밋으로 정확한 처리 시점 제어 가능 (권장)
  - max.poll.records
    - 한 번의 poll() 호출로 가져올 수 있는 최대 메시지 개수
    - 값이 크면 처리량 증가하지만, 처리 시간이 길어져 rebalance 위험이 커질 수 있음
  - auto.commit.interval.ms
    - enable.auto.commit=true일 때 오프셋을 커밋하는 주기
    - 값이 짧으면 커밋 빈도가 높아져 안정성 증가, 길면 성능은 좋지만 유실 가능성 증가
  - listener.ack-mode
    - (Spring Kafka) 오프셋 커밋 전략 설정
    - RECORD: 메시지 단위로 커밋 (안정성 ↑, 성능 ↓)
    - BATCH: poll 단위로 커밋 (기본값, 성능/안정성 균형)
    - MANUAL / MANUAL_IMMEDIATE: 개발자가 직접 커밋 시점 제어 (정확한 처리 보장에 유리)
- Broker or Topic
  - replication.factor
    - 각 파티션의 복제본 개수
    - 1: 성능은 좋지만 장애 시 데이터 유실 가능
    - 3: 일반적인 운영 환경 권장 (1개 장애까지 허용)
  - log.retention.ms / log.retention.bytes
    - 메시지 보관 기간(ms) 또는 최대 저장 용량(bytes)
    - 시간/용량 중 하나라도 초과하면 오래된 메시지부터 삭제
    - 값이 크면 데이터 보존은 유리하지만 디스크 사용량 증가
  - message.max.bytes
    - 브로커가 허용하는 최대 메시지 크기
    - 이 값을 초과하면 프로듀서 전송 실패 발생
    - 큰 메시지를 처리하려면 프로듀서(max.request.size), 컨슈머(fetch.max.bytes) 설정도 함께 조정 필요
    
### 컨슈머에서 발생할 수 있는 메시지 중복 처리
- 중복 이벤트가 발생할 수 있는 경우
  - 가정
    - enable.auto.commit = true
    - auto.commit.interval.ms = 5000ms
    - max.poll.records = 10
  - 발생 케이스
    - 현재 컨슈머가 처리중인 메시지의 offset = 21 ~ 30 (max.poll.records = 10 이기 때문에 10개씩 polling)
      - 이전까지 커밋 : CURRENT-OFFSET = 11
    - 23번째 offset를 처리하다가 컨슈머가 갑자기 종료된다면?
      - 아직 auto commit 주기가 도달하지 않았기 때문에 offset 21~30은 커밋되지 않은 상태
      - 재시작 시 마지막 커밋 지점(11)부터 다시 읽기 때문에 21~22는 물론, 23 이후 메시지까지 중복 처리 발생
- 해결방법
  - listener.ack-mode를 RECORD로 설정하여 레코드 당 1회 커밋하도록 설정
    - 각 메시지 처리 직후 커밋되므로 장애 시 중복 범위를 최소화할 수 있음
    - 단점 : 커밋 호출이 많아져 성능 저하 및 브로커 부하 증가
  - 메시지 고유 ID를 활용한 중복 처리 방지
    - 메시지에 유니크 키를 부여하고, 이미 처리된 ID를 저장/조회하여 중복 실행을 방지함 (DB, Redis 등 활용)
  - 비즈니스 로직 자체를 멱등적으로 설계
    - 동일 메시지가 여러 번 처리되더라도 결과가 동일하도록 설계 (예: 상태 기반 업데이트, upsert 등)
  - 버저닝을 활용한 멱등성 구현
    - 엔티티에 @Version(낙관적 락)을 적용하여 데이터 변경 시 버전을 증가시키고, 동일한 데이터에 대한 중복 처리를 제어할 수 있음
    - 이미 처리된 이벤트라면 버전이 증가된 상태이기 때문에, 동일 이벤트가 다시 들어와도 업데이트가 반영되지 않도록 구성 가능
    - 즉, 버전을 기준으로 “이미 반영된 변경인지”를 판단하여 중복 처리나 역순 처리 문제를 방지하는 방식으로 활용할 수 있음

### 트랜잭션과 @Transactional
- 트랜잭션이란?
  - 데이터베이스 작업을 하나의 논리적 단위로 묶어 모두 성공하거나 모두 실패하도록 보장하는 개념 (원자성, 일관성, 격리성, 지속성 - ACID)
- @Transactional
  - propagation
    - 트랜잭션 전파 옵션으로, 기존 트랜잭션이 있을 때 참여할지 새로 생성할지 결정
    - REQUIRED: 기존 트랜잭션 참여 (기본값)
    - REQUIRES_NEW: 항상 새로운 트랜잭션 생성 (기존 트랜잭션과 분리)
    - SUPPORTS: 트랜잭션이 있으면 참여, 없으면 비트랜잭션으로 실행
  - isolation
    - 트랜잭션 간 데이터 격리 수준을 설정
    - READ_UNCOMMITTED: 커밋되지 않은 데이터 조회 가능 (Dirty Read 발생)
    - READ_COMMITTED: 커밋된 데이터만 조회 가능
    - REPEATABLE_READ: 같은 조회 결과 보장 (Non-repeatable read 방지)
    - SERIALIZABLE: 가장 강력한 격리 수준 (성능 저하 가능)
  - timeout
    - 트랜잭션이 지정된 시간 내에 완료되지 않으면 롤백 처리
  - readOnly
    - 읽기 전용 트랜잭션으로 설정 시, 변경 감지(Dirty Checking)를 생략하여 성능 최적화 가능
  - rollbackFor
    - 지정한 예외 발생 시 롤백하도록 설정 (기본적으로 RuntimeException만 롤백)
  - noRollbackFor
    - 지정한 예외 발생 시 롤백하지 않도록 설정
  - value / transactionManager
    - 사용할 트랜잭션 매니저를 지정 (멀티 DB 환경에서 활용)

### 데이터베이스 동시성 문제
- 데이터베이스에서 발생할 수 있는 동시성 문제
  - 데이터 일관성이 깨지는 케이스
    - Dirty Read
      - 다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽는 현상으로, 이후 롤백되면 잘못된 데이터를 기반으로 로직이 수행될 수 있음
    - Non-Repeatable Read
      - 같은 트랜잭션 내에서 동일한 조회를 두 번 수행했을 때, 다른 트랜잭션의 커밋으로 인해 결과가 달라지는 현상
    - Phantom Read
      - 같은 조건으로 조회했을 때, 다른 트랜잭션의 INSERT/DELETE로 인해 결과 행의 개수가 달라지는 현상
    - Lost Update
      - 두 트랜잭션이 동일 데이터를 수정할 때, 먼저 반영된 변경이 이후 트랜잭션에 의해 덮어써지는 문제
  - 실행 중단
    - Deadlock
      - 두 개 이상의 트랜잭션이 서로의 자원을 점유한 채 상대방의 자원을 기다리며 무한 대기 상태에 빠지는 현상
    - Livelock
      - 트랜잭션들이 서로 양보하며 계속 상태를 변경하지만 실제 작업은 진행되지 않는 상태
- 트랜잭션 격리수준
  - READ_UNCOMMITTED
    - 커밋되지 않은 데이터도 조회 가능
    - Dirty Read, Non-Repeatable Read, Phantom Read 모두 발생 가능
  - READ_COMMITTED
    - 커밋된 데이터만 조회 가능
    - Dirty Read 방지
    - Non-Repeatable Read, Phantom Read는 여전히 발생 가능
  - REPEATABLE_READ
    - 동일 트랜잭션 내에서 같은 조회 결과를 보장
    - Dirty Read, Non-Repeatable Read 방지
    - Phantom Read는 발생 가능 (DB 구현에 따라 일부 방지되기도 함, 예: InnoDB)
  - SERIALIZABLE
    - 트랜잭션을 순차적으로 실행한 것과 동일한 결과 보장
    - Dirty Read, Non-Repeatable Read, Phantom Read 모두 방지
    - 대신 성능 저하 및 락 경쟁 증가

### 트랜잭셔널 아웃박스 (Transactional Outbox) 패턴
- 개념
  - 데이터베이스 변경과 메시지 발행을 하나의 트랜잭션으로 묶어, 데이터 정합성과 메시지 유실 방지를 동시에 보장하는 패턴
  - DB에는 반영됐지만 메시지 발행은 실패하는 이중 쓰기 문제를 해결하기 위해 사용됨
- 동작 방식
  - 비즈니스 로직 수행 + 이벤트를 Outbox 테이블에 함께 저장 (같은 트랜잭션)
  - 트랜잭션 커밋 이후, 별도의 프로세스가 Outbox 데이터를 읽어 메시지 브로커로 전송
  - 전송 완료 후 상태 변경 또는 삭제로 재처리 방지
- Polling 방식
  - @Scheduled 등을 활용하여 일정 주기마다 Outbox 테이블에서 미전송 데이터 조회 후 메시지 전송
  - 구현은 단순하지만 멀티 인스턴스 환경에서 문제 발생 가능
    - 여러 인스턴스가 동시에 동일 데이터를 조회하여 중복 전송 발생 가능
  - 보완 방법
    - 상태 컬럼 기반 처리 (READY → PROCESSING → SENT)
      - 조회 시 PROCESSING으로 변경하여 중복 처리 방지
    - SELECT ... FOR UPDATE
      - 트랜잭션 내에서 row lock을 통해 동시 접근 제어
    - 분산 락 (Redis 등)
      - 하나의 인스턴스만 Polling 수행하도록 제어
- CDC 기반 방식
  - DB의 변경 로그(binlog 등)를 기반으로 이벤트를 감지하여 메시지 브로커로 전달
  - Debezium과 같은 CDC 도구가 Outbox 테이블 변경을 감지하여 Kafka로 전송
  - 애플리케이션은 Outbox에 insert만 수행하고 메시지 발행은 CDC가 담당
- CDC를 사용하는 이유
  - Polling 방식의 멀티 인스턴스 중복 처리 문제 해결
    - 이벤트를 단일 스트림으로 처리하여 중복 전송 방지
  - 실시간성 확보
    - Polling 주기 없이 변경 즉시 이벤트 전파 가능
  - 애플리케이션과 메시지 브로커 간 결합도 감소
- 특징 및 고려사항
  - Outbox 패턴은 기본적으로 At-least-once 전달을 보장하므로, 컨슈머 측 멱등성 설계 필요
  - Outbox 테이블 데이터 증가에 대한 정리 전략 필요 (TTL, 배치 삭제 등)
  - 초기에는 Polling으로 시작하고, 트래픽 증가 시 CDC로 전환하는 전략이 일반적

### 보상 트랜잭션 (Compensating Transaction) 패턴
- 보상 트랜잭션의 의미
  - 분산 환경에서 하나의 트랜잭션을 여러 서비스에 걸쳐 수행할 때, 중간에 실패가 발생하면 이전 단계에서 수행된 작업을 “되돌리는(취소하는)” 별도의 트랜잭션을 수행하는 방식
  - DB의 롤백처럼 자동으로 되돌릴 수 없기 때문에, 각 단계마다 “취소 로직”을 명시적으로 구현해야 함
  - 예: 주문 생성 → 결제 → 배송 중 결제 실패 시, 이미 생성된 주문을 취소하는 작업 수행
- 보상 트랜잭션과 사가(Saga)의 관계
  - 사가 패턴은 여러 개의 로컬 트랜잭션을 순차적으로 실행하고, 실패 시 보상 트랜잭션을 통해 이전 작업을 되돌리는 구조
  - 즉, 보상 트랜잭션은 사가 패턴을 구성하는 핵심 요소로, 각 단계마다 대응되는 “undo 작업”이라고 볼 수 있음
- 사가(Saga) 패턴
  - 하나의 큰 분산 트랜잭션을 여러 개의 로컬 트랜잭션으로 나누어 처리하고, 각 단계 성공 시 다음 단계로 진행
  - 중간에 실패하면 이미 수행된 단계들에 대해 보상 트랜잭션을 실행하여 데이터 정합성을 맞춤
  - 분산 트랜잭션(2PC)의 대안으로, 느슨한 결합과 높은 확장성을 제공
- 사가 패턴 구현 방식
  - 오케스트레이션 기반 (Orchestration)
    - 중앙에 오케스트레이터(조정자)가 존재하여 각 서비스의 호출 순서를 제어
    - 흐름
      - 오케스트레이터 → 서비스 A → 서비스 B → 서비스 C 순으로 명령 전달
      - 특정 단계 실패 시 오케스트레이터가 이전 서비스들에 보상 트랜잭션 요청
    - 특징
      - 흐름이 명확하고 제어가 쉬움
      - 중앙 컴포넌트에 의존성이 집중됨 (단일 장애 지점 가능)
  - 코레오그래피 기반 (Choreography)
    - 중앙 제어 없이 각 서비스가 이벤트를 발행하고, 이를 구독하여 다음 작업을 수행
    - 흐름
      - 서비스 A 완료 → 이벤트 발행 → 서비스 B가 구독 후 처리 → 이벤트 발행 → 서비스 C 처리
      - 실패 시 각 서비스가 실패 이벤트를 발행하고, 이를 기반으로 보상 트랜잭션 수행
    - 특징
      - 서비스 간 결합도가 낮고 확장성이 뛰어남
      - 전체 흐름을 파악하기 어렵고, 디버깅 및 관리가 복잡해질 수 있음
- 정리
  - 보상 트랜잭션은 “되돌리는 로직”
  - 사가는 “그걸 포함한 전체 흐름 관리 방식”
  - 오케스트레이션 vs 코레오그래피는 “흐름을 중앙에서 제어할지, 이벤트로 분산할지”의 차이

### 강력한 일관성 vs 최종적 일관성
- 강력한 일관성
  - 데이터에 변경이 발생했을 때, 시스템의 모든 노드에서 즉시 동일한 최신 데이터가 보이도록 보장하는 모델
  - 강력한 일관성만으로 시스템을 설계할 수 없는 이유
    - 분산된 서비스, 분산된 데이터
      - 네트워크 지연과 장애로 인해 모든 노드를 즉시 일치시키는 것이 현실적으로 어려움
    - 성능과 확장성의 한계
      - 동기화 비용 증가로 응답 지연이 커지고, 트래픽 증가 시 병목이 쉽게 발생함
- 최종적 일관성
  - 데이터 변경 후 시스템의 모든 노드에서 일시적으로 데이터가 다를 수 있지만, 결국에는 모든 노드가 동일한 데이터 상태를 갖게 되는 것을 보장하는 모델
- 사용자 경험 관점에서 선택
  - 강력한 일관성이 적절한 사례
    - 은행 계좌이체
    - 선착순 쿠폰 이벤트
    - 공연 티켓 예매
  - 최종적 일관성이 적절한 사례
    - 유튜브 영상 업로드
    - 조회수 집계
    - SNS 좋아요
    - 상품평 작성
    - 주문 완료 알림
- 정리
  - 절대 놓쳐서는 안되는 핵심 비즈니스 로직은 '강력한 일관성'으로 지켜내고,
  - 나머지 부가적인 기능들은 '최종적 일관성'으로 풀어내어
  - 성능과 안정성, 확장성이라는 세 마리 토끼를 모두 잡는 것이 중요함
