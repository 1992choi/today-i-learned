### 서비스 내부에서 애플리케이션을 나누는 이유
- 장점
  - 장애 격리
    - 하나의 서비스 장애가 전체 시스템으로 확산되지 않고, 영향 범위를 최소화할 수 있음
  - 독립적인 배포와 스케일링
    - 서비스별로 트래픽에 맞춰 개별 확장 및 배포가 가능해 효율적인 운영이 가능함
  - 각각의 애플리케이션을 서로 다른 팀에서 개발 가능
    - 팀 간 의존도를 낮추고 병렬 개발이 가능해 개발 속도를 높일 수 있음
  - 기술 스택, 언어 선택이 자유로움
    - 서비스 특성에 맞는 최적의 기술을 선택하여 성능과 생산성을 동시에 확보할 수 있음
- 단점
  - 복잡성, 운영비용 증가
    - 서비스 간 통신, 배포, 모니터링 등 관리 포인트가 늘어나 운영 난이도가 상승함
  - 데이터 일관성 문제
    - 분산된 데이터 관리로 인해 트랜잭션 처리와 일관성 유지가 어려워질 수 있음

### 이중화와 복제(Replication)
- 이중화와 복제(Replication)
  - 이중화와 복제는 견고한 시스템 설계의 기초이다.
    - 이중화
      - 동일한 역할을 수행하는 시스템을 여러 개 구성하여 하나에 장애가 발생해도 다른 시스템이 즉시 대체할 수 있도록 하는 방식
    - 복제
      - 데이터를 여러 시스템에 동일하게 복사해 저장하고, 장애 시 데이터 유실을 방지하거나 읽기 부하를 분산시키는 방식
- 이중화의 목적
  - 동일한 기능을 하는 시스템 구성 요소를 2개 이상 두어 단일 장애점(SPOF)를 제거하고, 일부 구성요소의 장애시에도 서비스가 지속될 수 있도록 만드는 방법
  - 장애 발생 시에도 트래픽을 정상 인스턴스로 우회하여 서비스 중단을 최소화하는 것이 핵심
- 실무에서의 이중화와 복제
  - 이중화 예시
    - 로드밸런서 + 애플리케이션
      - 여러 애플리케이션 인스턴스를 두고 로드밸런서가 트래픽을 분산하며, 특정 인스턴스 장애 시 자동으로 제외하여 가용성을 확보함
  - 복제 예시
    - DB
      - DB는 상태를 가지기 때문에 단순 이중화가 어렵고, 데이터 일관성을 유지하기 위해 복제 방식을 사용함
      - Primary(Write)는 쓰기를 담당하고, Replica(Read)는 복제된 데이터를 기반으로 조회를 처리하여 부하를 분산함
      - 복제 방식에 따라 Primary의 변경 사항을 Replica가 즉시 반영하지 않을 수 있으며, 동기 복제는 일관성을 보장하지만 지연이 발생하고 비동기 복제는 성능은 좋지만 데이터 불일치 가능성이 존재하는 트레이드오프가 있음
