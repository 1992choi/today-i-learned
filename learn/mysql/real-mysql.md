## 학습정리
### CHAR vs VARCHAR
- 공통점
  - 문자열 저장용 컬럼
  - 최대 저장 가능 문자 길이 명시 (바이트 수가 아님)
  - 문자 길이 기준이며 실제 저장 바이트는 문자셋(Character Set)에 따라 달라질 수 있음
- 차이점
  - 값의 실제 크기에 관계없이 고정된 공간 할당 여부
    - CHAR
      - 고정 길이(Fixed Length) 타입
      - 선언된 길이만큼 항상 동일한 공간을 사용
    - VARCHAR
      - 가변 길이(Variable Length) 타입
      - 실제 데이터 길이에 따라 사용 공간이 달라짐
  - 최대 저장 길이
    - CHAR(255)
      - 최대 255 문자 저장 가능
    - VARCHAR(16383)
      - 최대 약 16383 문자 저장 가능
      - 단, Row 전체 크기 제한의 영향을 받음
  - 저장된 값의 길이 관리 여부
    - VARCHAR
      - 가변 길이 타입이기 때문에 실제 문자열 길이를 저장하는 값이 별도로 존재
      - 일반적으로 문자열 앞에 1~2바이트 길이 정보가 함께 저장됨
    - CHAR
      - 기본적으로 길이 정보를 저장하지 않음
      - 항상 고정된 길이를 사용하기 때문
      - 단, UTF-8 같은 가변 길이 문자셋을 사용하는 경우 내부적으로 길이 관리가 필요할 수 있음
- CHAR 저장방식
  - 예: column1 이 CHAR(10) 컬럼이고 값으로 "ABCD"가 저장되는 경우
    - 실제 문자 데이터
      - ABCD
      - 4바이트 사용
    - 남은 공간
      - 6바이트
      - 공백 문자(space)로 채워짐
    - 최종 저장 형태
      - ABCD + 공백 6바이트
    - 즉 실제 데이터 길이가 4여도
      - 항상 10바이트 공간을 사용함
- VARCHAR 저장방식
  - 예: column1 이 VARCHAR(10) 컬럼이고 값으로 "ABCD"가 저장되는 경우
    - 길이 정보
      - "4글자"라는 정보를 저장
      - 1바이트 사용
    - 실제 문자 데이터
      - ABCD
      - 4바이트 사용
    - 최종 저장 형태
      - 길이 정보 1바이트
      - 실제 문자 4바이트
- CHAR 타입의 공간 낭비
  - 가변폭이 큰 경우, 공간낭비가 심함
    - 예
      - CHAR(100) 컬럼 존재
    - 실제 저장되는 데이터 길이
      - 어떤 값은 10자리
      - 어떤 값은 100자리
    - 만약 10자리 데이터가 저장된다면
      - 실제 데이터
        - 10바이트
      - 남는 공간
        - 90바이트
    - 결과
      - 90바이트 공간이 사용되지 않고 낭비됨
- 컬럼 값의 길이 변경시 작동 방법
  - VARCHAR(10) 컬럼에 기존 값이 "ABCD"라고 가정
  - 이후 값이 "ABCDEF"로 변경되는 상황
    - 기존 레코드
      - 4바이트 데이터 공간 사용
    - 변경 후 데이터
      - 6바이트 공간 필요
    - 문제
      - 기존 레코드가 저장된 위치에는
      - 추가로 사용할 공간이 없을 수 있음
    - 이 경우 동작 방식
      - 기존 위치에 데이터를 저장하지 못함
      - 새로운 충분한 공간을 가진 위치에 레코드를 다시 저장
      - 기존 위치에는
        - 삭제 표시 또는 포인터 정보가 남음
    - 이 과정에서 데이터가 여러 위치에 흩어지게 됨
      - 이를 조각화(Fragmentation)라고 함
        - 데이터가 물리적으로 분산되어 저장되는 현상
      - 조각화가 발생하면
        - 데이터가 연속적으로 저장되지 않음
        - 조회 시 여러 위치를 읽어야 함
        - 디스크 I/O 증가
        - 조회 성능 저하 발생
- VARCHAR보다 CHAR 를 선택해야하는 경우
  - 값의 가변 길이 범위 폭이 좁고
  - 자주 변경되는 경우 (특히 인덱스된 컬럼인 경우)
    - CHAR 특징
      - 고정 길이 구조
      - 값이 변경되어도 레코드 크기가 변하지 않음
      - 따라서
        - 기존 위치에서 바로 업데이트 가능
        - 레코드 이동이 발생하지 않음
    - VARCHAR 특징
      - 값 길이가 증가하면
        - 기존 공간에 저장하지 못할 수 있음
      - 이 경우
        - 새로운 위치에 레코드를 다시 저장해야 함
      - 특히 인덱스가 존재하는 경우
        - 레코드 위치가 변경됨
        - 인덱스 포인터도 함께 수정 필요
    - 결과적으로
      - 디스크 I/O 증가
      - 인덱스 업데이트 비용 증가
      - 조각화 발생 가능성 증가
