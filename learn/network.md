## 개념 정리
### Layer와 Layered 구조의 기본 개념
- Layer란?
  - 레이어(Layer)는 네트워크 통신 기능을 수행하는 기능 단위이다.
  - 하나의 Layer는 특정 기능만을 책임지는 단위이다.
- Layered 구조란?
  - 네트워크 기능을 여러 계층으로 나누어 구성한 구조이다.
  - 하위 계층의 기능을 기반으로 상위 계층이 동작하는 방식이다.
  - 각 계층은 의존 관계로 연결된 구조이다.
    - 상위 계층은 반드시 바로 아래 계층이 제공하는 서비스를 사용하는 구조이다.
    - 예를 들어, L4는 L3, L2, L1이 정상 동작해야 의미를 가지는 구조이다.
- 네트워크에서 Layered 구조가 왜 중요한가?
  - 복잡한 통신을 기능별로 분리하는 목적이다.
  - 모듈화를 위한 구조이다.
  - 표준화된 통신 규칙 적용을 위한 구조이다.

### 네트워크와 네트워킹 그리고 개념
- Network란?
  - Network는 관계의 구조이다.
    - 장치와 장치가 서로 연결되어 관계를 맺고 있는 상태이다.
  - 단순한 연결 선을 의미하는 것이 아니라, 연결을 통해 형성된 구조적 관계를 의미하는 개념이다.
- Networking이란?
  - Networking은 상호작용을 만드는 과정이다.
    - 연결된 장치들이 실제로 데이터를 주고받을 수 있도록 만드는 행위이다.
  - 단순히 연결하는 것이 아니라, 통신이 가능하도록 설정하고 구성하는 활동이다.

### User mode와 Kernel mode
- User mode란?
  - 제한된 권한으로 동작하는 실행 모드이다.
  - 일반 애플리케이션 프로그램이 실행되는 영역이다.
  - 하드웨어 자원에 직접 접근할 수 없는 구조이다.
  - 네트워크 요청을 생성하는 주체가 되는 영역이다.
  - 시스템 콜을 통해서만 Kernel mode에 작업을 요청할 수 있는 구조이다.
- Kernel mode란?
  - 모든 시스템 자원에 접근 가능한 특권 모드이다.
  - 운영체제가 실행되는 영역이다.
  - 메모리, 디스크, 네트워크 장치에 직접 접근할 수 있는 구조이다.
  - 네트워크 스택이 동작하는 핵심 영역이다.
  - User mode의 요청을 받아 실제 처리를 수행하는 구조이다.
- OSI 7계층과 HW / Kernel / User 매핑
  - OSI 7계층은 통신 기능을 나눈 논리적 개념 모델이다.
  - OSI 7계층은 통신 기능을 나눈 논리적 개념 모델이며, HW / Kernel / User 구분은 실제 구현 위치 기준의 구조이지만, 구현 관점에서 다음과 같이 매핑하여 설명할 수 있다.
    - HW 영역
      - OSI 1계층 (Physical)이 해당하는 영역이다.
      - OSI 2계층 일부가 포함되는 구조이다.
        - NIC에서 처리되는 MAC 동작
        - CRC 검사 등 일부 프레임 처리 기능
    - Kernel mode 영역
      - OSI 2계층 일부가 포함되는 구조이다.
      - OSI 3계층 (IP)이 동작하는 영역이다.
      - OSI 4계층 (TCP/UDP)이 동작하는 영역이다.
    - User mode 영역
      - OSI 5 ~ 7계층이 동작하는 영역이다.
      - HTTP, FTP, DNS 등 애플리케이션 프로토콜이 구현되는 영역이다.
      - 브라우저, 서버 프로그램 등 응용 프로그램이 실행되는 구조이다.

### OSI 7계층 주요 프로토콜 정리
- 1계층 (Physical Layer, 물리계층)
  - 전기 신호, 전압, 케이블, 커넥터 등 실제 비트를 전송하는 물리적 매체를 다루는 계층이다.
  - 데이터를 0과 1의 신호 형태로 전달하는 역할을 수행하는 구조이다.
- 2계층 (Data Link Layer, 데이터 링크 계층)
  - Ethernet이 대표적인 프로토콜이다.
  - 같은 네트워크(LAN) 내에서 프레임 단위로 데이터를 전달하는 계층이다.
  - 식별자 : MAC 주소이다.
    - LAN 카드(NIC)를 식별하는 물리적 주소이다.
    - 동일 네트워크 내에서 목적지를 구분하는 기준이 되는 주소이다.
- 3계층 (Network Layer, 네트워크 계층)
  - Internet 계층에 해당하는 영역이다.
    - IP가 대표적인 프로토콜이다.
    - Internet 외에도 다른 프로토콜이 존재하지만, 일반적으로 IP 중심으로 설명하는 구조이다.
  - 서로 다른 네트워크 간 패킷을 전달하는 계층이다.
  - 라우팅을 통해 경로를 결정하는 역할을 수행하는 구조이다.
  - 식별자 : IP 주소이다.
    - 호스트를 식별하는 논리적 주소이다.
    - 네트워크 상에서 목적지를 구분하는 기준이 되는 주소이다.
- 4계층 (Transport Layer, 전송 계층)
  - TCP, UDP 프로토콜이 해당하는 계층이다.
  - 종단 간 통신을 담당하는 계층이다.
  - 데이터의 신뢰성, 흐름 제어, 오류 제어 등을 처리하는 구조이다.
  - 식별자 : Port 번호이다.
    - 하나의 호스트 내부에서 어떤 프로세스와 통신할지 구분하는 식별자이다.
- 5계층 (Session Layer, 세션 계층)
  - SSL(TLS)이 해당하는 영역으로 설명하는 경우가 많다.
  - 통신 세션을 설정하고 유지하는 역할을 수행하는 계층이다.
  - 암호화된 세션을 생성하여 보안을 제공하는 구조이다.
- 6계층 (Presentation Layer, 표현 계층)
  - 데이터 형식 변환, 인코딩/디코딩, 압축, 암호화 등을 담당하는 계층이다.
  - 서로 다른 시스템 간 데이터 표현 방식을 맞추는 역할을 수행하는 구조이다.
- 7계층 (Application Layer, 응용 계층)
  - HTTP가 대표적인 프로토콜이다.
  - 사용자가 직접 사용하는 애플리케이션 프로토콜이 동작하는 계층이다.
  - 웹 브라우저와 웹 서버 간 요청과 응답이 이루어지는 구조이다.

### Host는 이렇게 외우자
- Host란?
  - 간단하게 표현하면, 네트워크에 연결된 computer를 지칭하는 개념이다.
  - 네트워크 상에서 IP 주소를 가지고 통신에 참여하는 장치를 의미하는 용어이다.
  - 데이터를 생성하거나 소비하는 주체가 되는 장치이다.
  - 서버, PC, 스마트폰 등 실제 통신의 끝단에 위치하는 장치가 이에 해당하는 구조이다.

### 스위치가 하는 일과 비용
- 패킷이란?
  - 네트워크에서 전송되는 데이터의 단위이다.
  - 데이터에 출발지/목적지 정보가 포함된 구조이다.
  - 네트워크 장비는 이 정보를 기준으로 데이터를 전달한다.
- Switch란?
  - 네트워크 내부에서 데이터를 목적지 방향으로 전달하는 장치이다.
  - 기본적으로 2계층에서 동작하며, 전달 대상은 프레임이다.
- 역할
  - 수신한 데이터를 분석한다.
  - 목적지 정보를 기준으로 다음 전달 지점을 결정한다.
  - 데이터를 생성하지 않고 중간에서 전달만 수행한다.
- 경로를 선택하는 근거
  - 계층에 따라 기준이 다르다.
    - 2계층 장비: MAC 주소를 기준으로 프레임을 전달한다.
    - 3계층 장비: IP 주소를 기준으로 패킷의 경로를 결정한다.
    - 4계층 이상 장비: 포트 번호 등 추가 정보를 활용하기도 한다.
  - 즉, 어떤 정보를 기준으로 판단하느냐는 장비가 동작하는 계층에 따라 달라진다.
- L3 스위치와 라우터
  - L3 스위치는 3계층(Network Layer)에서 IP 주소를 기준으로 패킷의 경로를 결정한다.
  - 이러한 동작은 라우팅(Routing) 기능과 동일하다.
  - 따라서 기능 관점에서는 L3 스위치를 라우터로 볼 수 있다.
  - 다만, 장비 분류상으로는 스위치 계열 장비에 라우팅 기능이 추가된 형태이다.

### NIC, L2 Frame, LAN 카드 그리고 MAC 주소
- NIC (Network Interface Card)
  - 흔히 LAN 카드라고 부르는 하드웨어 장치이다.
  - 컴퓨터를 네트워크에 연결해주는 물리적 인터페이스이다.
  - NIC은 하드웨어 장치이며, 고유한 MAC 주소를 가진다.
- MAC 주소
  - 2계층(Data Link Layer)에서 사용되는 물리적 주소이다.
  - 일반적으로 48비트(6바이트)로 구성된다.
  - 동일 네트워크(LAN) 내부에서 장치를 식별하는 용도로 사용된다.
  - NIC에 할당되어 출고되며, 필요에 따라 소프트웨어적으로 변경이 가능하다.
- L2 Frame
  - 2계층에서 전송되는 데이터 단위이다.
  - 구성 요소
    - 목적지 MAC 주소, 출발지 MAC 주소, 데이터, 오류 검출 정보 등
  - 동일 네트워크 내에서 스위치가 전달하는 대상이다.

### L2 스위치에 대해서
- L2 스위치란?
  - OSI 2계층(Data Link Layer)에서 동작하는 장비이다.
  - MAC 주소를 기준으로 프레임을 전달한다.
  - 동일 네트워크(LAN) 내부에서 장비 간 통신을 가능하게 한다.
- L2 스위치의 역할 구분
  - L2 스위치는 네트워크 설계 구조에 따라 역할이 나뉜다.
  - 대표적으로 Access와 Distribution 역할로 구분된다.
  - 이는 장비 종류의 차이가 아니라, 배치 위치와 기능 중심의 구분이다.
- L2 Access Switch
  - End-point와 직접 연결되는 위치에 배치된다.
  - MAC 주소를 기준으로 프레임을 스위칭한다.
  - 단말이 네트워크에 접속하는 지점 역할을 수행한다.
  - 스위치와 장비가 정상적으로 연결되어 통신 가능한 상태를 Link-up이라고 한다.
    - 물리적 연결이 끊기거나 통신이 불가능한 상태는 Link-down이라고 한다.
- L2 Distribution Switch
  - 여러 L2 Access 스위치를 상위에서 연결하는 위치에 배치된다.
  - 트래픽을 집선(Aggregation)하는 역할을 수행한다.
  - VLAN(Virtual LAN) 기능을 제공하는 것이 일반적이다.
    - VLAN(Virtual LAN)이란?
      - 하나의 물리적 네트워크를 논리적으로 여러 개의 네트워크처럼 분리하는 기능이다.
      - 서로 다른 VLAN에 속한 장비는 기본적으로 2계층에서 직접 통신할 수 없다.
      - 브로드캐스트 도메인을 분리하는 목적을 가진다.

### LAN과 WAN의 경계 그리고 Broadcast
- Broadcast
  - Broadcast란?
    - 개념
      - Broadcast는 네트워크에 연결된 모든 장비에게 데이터를 동시에 전달하는 통신 방식이다.
      - 특정 대상이 아니라 같은 네트워크에 속한 모든 장비에게 패킷을 전달하는 것이 특징이다.
  - Broadcast의 주소
    - 특별한 주소가 존재함
    - MAC과 IP 모두 존재
      - MAC 주소: FF:FF:FF:FF:FF:FF
      - IP 주소: 네트워크의 Broadcast 주소 (예: 192.168.0.255)
  - 특징
    - 하나의 장비(Host)가 Broadcast를 전송하면 같은 네트워크에 있는 모든 장비가 해당 패킷을 수신한다.
    - Broadcast 트래픽이 많아지면 네트워크 전체 성능에 영향을 줄 수 있다.
    - 따라서 Broadcast의 범위(Broadcast Domain)는 가능한 작게 유지하는 것이 좋다.
    - 하지만 Broadcast 범위를 너무 작게 나누면 필요한 장비까지 패킷이 전달되지 않을 수 있다.
    - 이러한 이유로 네트워크에서는 보통 IP 네트워크 단위(Subnet)를 기준으로 Broadcast 범위를 구분한다.
- LAN과 WAN
  - 강의에서의 설명
    - HW는 물리(Physical) 영역
    - SW는 논리(Logical) 영역으로 표현하기도 한다.
  - LAN
    - 물리적 통신 범위라고 설명하기도 한다.
    - 하나의 건물, 사무실, 데이터센터처럼 물리적으로 가까운 네트워크를 의미한다.
  - WAN
    - 논리적 통신 범위라고 설명하기도 한다.
    - 여러 LAN을 논리적으로 연결한 더 넓은 네트워크를 의미한다.
    - 대표적으로 인터넷이나 통신사 네트워크를 통해 연결되는 구조가 이에 해당한다.
  - 주의할 점
    - LAN = 물리 / WAN = 논리 라는 구분이 공식적인 정의는 아니다.
    - 단지 네트워크 범위를 직관적으로 이해하기 위한 설명 방식이다.

### IPv4 주소의 기본 구조
- IPv4
  - 구조
    - 32bit로 구성된다.
      - = 8bit × 4
      - = 256개의 값 × 4
      - 각 옥텟은 0 ~ 255 범위를 가진다.
    - Ex) 192.168.0.10
      - 2진수(bit)로 표현하면 아래와 같다.
        - `1100 0000 . 1010 1000 . 0000 0000` . `0000 1010`
      - 이 예시에서는 앞 24bit가 Network ID이고, 뒤 8bit가 Host ID이다.
        - `1100 0000 . 1010 1000 . 0000 0000` → Network ID
        - `0000 1010` → Host ID

### L3 IP Packet으로 외워라
- Packet 이란?
  - 네트워크에서 전송되는 데이터의 단위이다.
  - 3계층(Network Layer)에서 다루는 데이터 단위를 Packet이라고 한다.
  - 상위 계층에서 전달된 데이터를 목적지까지 전달하기 위해 주소 정보 등을 포함하여 구성된 데이터 구조이다.
- 구성
  - Packet은 Header와 Payload로 나뉜다.
    - Header
      - 패킷을 전달하기 위해 필요한 제어 정보가 담기는 영역이다.
      - 대표적으로 다음과 같은 정보가 포함된다.
        - 출발지 IP 주소
        - 목적지 IP 주소
        - 프로토콜 정보(TCP, UDP 등)
        - TTL(Time To Live)
      - 네트워크 장비는 Header 정보를 기준으로 패킷을 전달한다.
    - Payload
      - 실제 전달하려는 데이터가 들어 있는 영역이다.
      - 상위 계층(예: TCP/UDP, HTTP 등)에서 전달된 데이터가 포함된다.
- MTU (Maximum Transmission Unit)
  - 한 번에 전송할 수 있는 최대 데이터 크기를 의미한다.
  - 네트워크 장비나 인터페이스가 처리할 수 있는 패킷의 최대 크기이다.
  - 일반적인 Ethernet 환경에서는 MTU가 1500 byte인 경우가 많다.
  - 패킷의 크기가 MTU보다 크면 여러 개의 패킷으로 나누어 전송될 수 있다.

### Encapsulation과 Decapsulation
- 데이터 포함 구조
  - 네트워크 데이터는 계층 구조로 감싸지는 형태로 구성된다.
  - 예를 들어 다음과 같은 포함 관계로 이루어진다.
    - `L2 Frame`은 Header와 Payload로 구성된다. Payload에는 L3 Packet이 포함된다.
      - Header
      - Payload
        - `L3 Packet` (마찬가지로 Payload에 Segment 존재)
          - Header
          - Payload
            - `L4 Segment`
              - Header
              - Payload
                - L5 ~ L7 데이터 (Application Data)
  - 즉, 상위 계층의 데이터가 하위 계층의 Payload 안에 포함되는 구조이다.
- Encapsulation
  - 데이터를 전송할 때 상위 계층 데이터에 하위 계층 Header가 하나씩 추가되는 과정을 의미한다.
  - 각 계층이 자신의 제어 정보를 추가하면서 데이터를 감싸는 구조이다.
  - 결과적으로 상위 계층 데이터가 하위 계층의 Payload 안에 포함된다.
- Decapsulation
  - 수신 측에서 하위 계층부터 Header를 하나씩 제거하면서 데이터를 상위 계층으로 전달하는 과정을 의미한다.
  - 즉, Encapsulation의 반대 과정이다.
  - 네트워크 장비나 호스트는 Header 정보를 확인한 뒤 제거하고 내부 데이터를 다음 계층으로 전달한다.

### 패킷의 생성과 전달
- 패킷은 데이터 통신 과정에서 생성되어 네트워크를 통해 전달되고, 목적지에 도착하면 원래 데이터로 복원된다.
  - 패킷의 생성
    - 사용자가 웹 요청, 메시지 전송 등의 작업을 하면 Application 계층에서 데이터가 생성된다.
    - 이 데이터는 네트워크 계층을 내려가면서 각 계층의 Header가 추가된다.
    - 이 과정을 Encapsulation이라 하며, 최종적으로 L3 Packet 형태가 만들어진다.
  - 패킷의 전달
    - 생성된 패킷은 네트워크 장비(스위치, 라우터 등)를 거쳐 목적지로 전달된다.
    - 라우터는 Packet의 IP 주소를 기반으로 다음 목적지를 결정하여 전달한다.
  - 패킷의 소멸
    - 패킷이 목적지 장치에 도착하면 Decapsulation 과정이 이루어진다.
    - 각 계층의 Header가 제거되면서 원래의 Application 데이터가 복원된다.
    - 최종적으로 해당 데이터는 Application에서 사용된다.

### 계층별 데이터 단위
- L2 (Data Link Layer)
  - 데이터 단위 : Frame
  - 실제 물리 네트워크에서 전달되는 단위이다.
- L3 (Network Layer)
  - 데이터 단위 : Packet
  - IP 주소를 기반으로 목적지까지 전달된다.
- L4 (Transport Layer)
  - 데이터 단위 : Segment
  - 상위 계층의 데이터를 일정 크기로 나누어 전송한다.
  - MSS(Maximum Segment Size)에 의해 한 Segment의 최대 데이터 크기가 결정된다.
- L5 이상 (Application 영역, User 모드)
  - 데이터 단위 : Stream
  - 사용자가 생성한 연속적인 데이터 흐름이다.
  - 이 데이터는 전송을 위해 L4에서 여러 Segment로 나뉘게 된다.
    - 이는 한 번에 전송할 수 있는 데이터 크기가 제한되어 있기 때문이다.
    - L4에서는 MSS(Maximum Segment Size)에 의해 하나의 Segment가 가질 수 있는 최대 데이터 크기가 결정된다.
    - 따라서 MSS보다 큰 데이터는 여러 개의 Segment로 나누어 전송된다.
    - 이러한 과정을 Segmentation이라고 한다.

### 이해하면 인생이 바뀌는 TCP/IP 송·수신 구조
- 구조
  - TCP/IP 통신 구조는 크게 소프트웨어 영역과 하드웨어 영역으로 나누어 볼 수 있다.
    - 소프트웨어
      - User Mode
        - Application이 동작하는 영역
        - 네트워크 통신은 Socket I/O를 통해 커널에 요청한다
      - Kernel Mode
        - 실제 네트워크 통신 처리가 이루어지는 영역
        - TCP
        - IP
    - 하드웨어
      - NIC(Network Interface Card) 등을 통해 실제 네트워크로 데이터가 전송된다.
- 과정
  - 가정
    - 예를 들어 A와 B 두 시스템이 있고, A에서 `profile.png` 파일을 B로 전송한다고 가정한다.
  - 송신 과정
    - Application(User Mode)에서 `profile.png` 파일을 읽는다.
    - 읽은 데이터는 Socket I/O를 통해 Kernel 영역으로 전달된다.
    - 전달된 데이터는 송신 버퍼(Send Buffer)에 저장된다.
    - 파일과 같은 데이터는 한 번에 모두 전송할 수 없기 때문에 일정 크기로 나누어 전송해야 한다.
    - 이때 데이터를 여러 단위로 나누는 과정을 Segmentation이라고 한다.
    - 나뉜 데이터는 TCP Segment 형태로 만들어지고 IP 계층을 거쳐 네트워크로 전송된다.
  - 수신 과정
    - B 시스템에서는 네트워크를 통해 TCP Segment를 수신한다.
    - 수신된 데이터는 Kernel 영역의 수신 버퍼(Receive Buffer)에 저장된다.
    - TCP는 데이터를 정상적으로 받았다는 신호로 ACK를 송신 측에 전송한다.
    - 송신 측은 ACK를 받으면 수신 측이 정상적으로 데이터를 받고 있다고 판단하고 다음 데이터를 계속 전송한다.
    - 수신 버퍼에 쌓인 데이터는 Application(User Mode)이 Socket을 통해 읽어간다.
  - 수신 버퍼와 흐름 제어
    - 수신 측의 Receive Buffer에는 전달받은 데이터가 계속 쌓이게 된다.
    - Application이 데이터를 읽어가는 속도가 느리면 버퍼의 여유 공간이 점점 줄어든다.
    - TCP는 현재 수신 버퍼의 여유 공간을 송신 측에 알려주어 전송 속도를 조절한다.
- 네트워크 통신 중 발생할 수 있는 문제
  - 유실 (Packet Loss)
    - 전송 중 패킷이 네트워크에서 사라지는 현상
  - 재전송 (Retransmission)
    - 송신 측이 일정 시간 동안 ACK를 받지 못하면 패킷이 유실되었다고 판단하고 다시 전송한다
  - DUP ACK
    - 동일한 ACK가 여러 번 도착하는 현상
  - Out of Order
    - 네트워크 경로 등의 이유로 패킷의 도착 순서가 뒤바뀌는 현상
  - Zero Window
    - 수신 측 버퍼에 더 이상 여유 공간이 없어 데이터를 받을 수 없는 상태
    - Application이 버퍼의 데이터를 처리하는 속도가 느릴 때 발생할 수 있다

### IP 헤더 형식
- IPv4 Header 형식
  - IPv4 Packet의 Header에는 패킷 전달을 위해 필요한 다양한 정보가 포함된다.
    - Version
      - IP 버전을 나타낸다.
      - IPv4의 경우 값은 4이다.
    - IHL (Internet Header Length)
      - IP Header의 길이를 나타낸다.
      - 기본 헤더는 20byte이며 옵션이 추가되면 길이가 증가할 수 있다.
    - Type of Service (TOS) / DSCP
      - 패킷의 우선순위나 서비스 품질(QoS)을 지정하기 위한 필드이다.
    - Total Length
      - IP Packet 전체 길이를 나타낸다.
      - Header와 Payload를 모두 포함한 크기이다.
    - Identification
      - 패킷이 분할(Fragmentation) 되었을 때 원래 패킷을 식별하기 위한 값이다.
    - Flags
      - Fragmentation과 관련된 제어 정보를 담는다.
      - 패킷 분할 여부 등을 표시한다.
    - Fragment Offset
      - 분할된 패킷이 원래 데이터에서 어느 위치에 있는지를 나타낸다.
    - TTL (Time To Live)
      - 패킷이 네트워크에서 살아있을 수 있는 최대 홉 수를 의미한다.
      - 라우터를 하나 지날 때마다 값이 감소한다.
    - Protocol
      - 상위 계층 프로토콜을 나타낸다.
      - 예: TCP, UDP, ICMP 등
    - Header Checksum
      - IP Header의 오류를 검증하기 위한 값이다.
    - Source IP Address
      - 송신자의 IP 주소
    - Destination IP Address
      - 목적지의 IP 주소
    - Options
      - 추가적인 기능을 위한 선택적 필드이다.
      - 일반적으로는 거의 사용되지 않는다.
    - Padding
      - Header 길이를 32bit 단위로 맞추기 위해 채워지는 값이다.

### 서브넷 마스크와 CIDR
- 서브넷 마스크
  - Subnet Mask
    - Subnet Mask는 IP 주소에서 Network 영역과 Host 영역을 구분하기 위한 값이다.
    - IP 주소는 32bit로 이루어져 있으며, 이 중 일부는 Network를 식별하고 나머지는 Host를 식별하는 데 사용된다.
    - Subnet Mask는 어떤 비트가 Network이고 어떤 비트가 Host인지를 표시하는 역할을 한다.
  - 예시
    - IP 주소
      - 192.168.0.10
    - IP를 2진수로 표현
      - 1100 0000 . 1010 1000 . 0000 0000 . 0000 1010
    - Subnet Mask
      - 1111 1111 . 1111 1111 . 1111 1111 . 0000 0000
    - IP와 Subnet Mask를 AND 연산
      - 1100 0000 . 1010 1000 . 0000 0000 . 0000 0000
  - 의미
    - Subnet Mask는 Network 영역을 1로 표시하고 Host 영역을 0으로 표시한다.
    - IP 주소와 Subnet Mask를 AND 연산하면 Network 주소를 얻을 수 있다.
    - 위 예시에서는 다음과 같이 해석된다.
      - Network 영역
        - 1100 0000 . 1010 1000 . 0000 0000
        - 192.168.0
      - Host 영역
        - 0000 1010
        - 10
    - 따라서 해당 IP는 192.168.0.0 네트워크에 속한 호스트라고 볼 수 있다.
- CIDR (Classless Inter-Domain Routing)
  - 개념
    - CIDR는 IP 주소에서 Network 영역의 길이를 비트 단위로 표현하는 방법이다.
    - Subnet Mask를 간단하게 표기하기 위해 사용된다.
  - 예시
    - 192.168.0.10 / 24
  - 의미
    - /24는 앞의 24bit가 Network 영역이라는 의미이다.
    - 즉 아래와 같은 Subnet Mask와 동일하다.
      - 11111111.11111111.11111111.00000000
      - 255.255.255.0
    - 따라서
      - IP : 192.168.0.10
      - Network : 192.168.0.0
      - Subnet Mask : 255.255.255.0
      - CIDR 표기 : /24
- 정리
  - Subnet Mask는 Network와 Host 영역을 구분하는 값이다.
  - IP와 Subnet Mask를 AND 연산하면 Network 주소를 구할 수 있다.
  - CIDR는 Subnet Mask를 비트 길이로 간단하게 표현하는 방법이다.

### Broadcast IP 주소
- Broadcast IP는 특정 네트워크에 속한 모든 Host에게 데이터를 전송할 때 사용하는 주소이다.
- 하나의 네트워크에서 Host 영역의 비트가 모두 1인 주소가 Broadcast 주소가 된다.
- 예시
  - IP 주소
    - 192.168.0.255
  - 2진수로 표현
    - 1100 0000 . 1010 1000 . 0000 0000 . 1111 1111
- 의미
  - 서브넷 마스크가 `/24` 라고 가정하면 다음과 같은 구조가 된다.
    - Network 영역
      - 1100 0000 . 1010 1000 . 0000 0000
      - 192.168.0
    - Host 영역
      - 1111 1111
      - 255
  - Host 영역이 모두 1이기 때문에 이 주소는 해당 네트워크의 Broadcast 주소가 된다.
- 동작 방식
  - 어떤 장비가 `192.168.0.255`로 패킷을 보내면, 같은 네트워크에 있는 모든 장비가 이 패킷을 수신하게 된다.

### Host 자신을 가리키는 IP 주소
- 127.0.0.1
  - 자기 자신의 시스템을 가리키는 IP 주소이다.
  - 이를 Loopback Address라고 한다.
- 개념
  - 네트워크 통신을 외부로 보내지 않고, 같은 시스템 내부에서 통신하기 위해 사용되는 주소이다.
  - 즉, 자신의 컴퓨터가 자기 자신과 통신할 때 사용된다.
- 동작 방식
  - 일반적인 IP 통신 흐름 (아래와 같이 상위 계층에서 하위 계층으로 데이터가 내려가며 네트워크로 전달된다.)
    - Application
    - TCP
    - IP
    - NIC (네트워크 인터페이스)
    - 실제 네트워크 전송
  - 하지만 `127.0.0.1`로 통신할 경우
    - Application
    - TCP
    - IP
    - Loopback Interface
    - 다시 자신의 시스템으로 전달
- 특징
  - `127.0.0.1` 주소는 IP 계층 아래의 실제 네트워크로 전달되지 않는다.
  - 즉 NIC나 물리 네트워크로 나가지 않는다.
  - 운영체제 내부의 Loopback 인터페이스를 통해 다시 자신의 시스템으로 전달된다.

### TTL과 단편화
- TTL (Time To Live)
  - IP Packet이 네트워크에서 무한히 떠돌지 않도록 하기 위한 값이다.
  - 패킷이 라우터를 하나 지날 때마다 TTL 값이 1씩 감소한다.
  - TTL 값이 0이 되면 해당 패킷은 폐기된다.
  - 이를 통해 라우팅 오류나 네트워크 루프가 발생했을 때 패킷이 계속 순환하는 것을 방지한다.
- 단편화 (Fragmentation)
  - 네트워크 장비가 한 번에 전송할 수 있는 최대 크기는 MTU(Maximum Transmission Unit)로 제한된다.
  - 전송하려는 IP Packet의 크기가 MTU보다 클 경우 패킷을 여러 개로 나누어 전송하게 된다.
  - 이렇게 하나의 패킷을 여러 개로 나누는 과정을 단편화(Fragmentation)라고 한다.
  - 단편화와 하향 평준화
    - 네트워크 경로에는 여러 장비와 링크가 존재하며 각 구간마다 MTU 크기가 다를 수 있다.
    - 이때 전송되는 패킷의 크기는 결국 경로 중 가장 작은 MTU에 맞추어 전달되어야 한다.
      - 현대 네트워크에서는 성능 문제 때문에 중간 라우터에서의 단편화를 가능한 피하는 것이 일반적이다. (즉 단편화를 하는 것보다 하향 평준화가 나을 수 있기 때문)
    - 즉, 네트워크 전체가 가장 작은 MTU 기준으로 맞춰지는 형태가 되는데 이를 하향 평준화라고 볼 수 있다.
- 단편의 조립 (Reassembly)
  - 단편화되어 전송된 패킷은 목적지에 도착한 후 다시 원래의 패킷으로 합쳐진다.
  - 이 과정은 일반적으로 수신 측 Host에서 수행된다.
  - IP Header의 Identification, Flags, Fragment Offset 정보를 이용하여 원래 패킷을 재구성한다.

### 인터넷 설정 자동화를 위한 DHCP
- 인터넷 사용 전에 필요한 기본 네트워크 설정
  - IP 주소
    - 네트워크에서 자신의 장치를 식별하기 위한 주소이다.
    - 각 장치는 고유한 IP 주소를 가져야 네트워크에서 통신이 가능하다.
  - Subnet Mask
    - IP 주소에서 Network 영역과 Host 영역을 구분하기 위한 값이다.
    - 이를 통해 같은 네트워크인지, 다른 네트워크인지 판단할 수 있다.
  - Gateway IP 주소
    - 다른 네트워크로 나가기 위한 출입구 역할을 하는 장비의 주소이다.
    - 보통 라우터의 IP 주소가 Gateway로 설정된다.
  - DNS 주소
    - 사람이 읽기 쉬운 도메인 이름을 IP 주소로 변환해주는 서버의 주소이다.
    - 예를 들어 `google.com` 같은 도메인을 실제 서버의 IP 주소로 변환해준다.
- DHCP (Dynamic Host Configuration Protocol)
  - 개념
    - DHCP는 네트워크에 접속하는 장치에게 IP 주소와 네트워크 설정을 자동으로 할당해주는 프로토콜이다.
    - 사용자는 수동으로 IP 주소, Subnet Mask, Gateway, DNS 등을 설정할 필요 없이 자동으로 네트워크 설정을 받을 수 있다.
  - DHCP로 자동 설정되는 정보
    - IP 주소
    - Subnet Mask
    - Gateway IP
    - DNS 주소
  - Broadcast와 DHCP
    - 네트워크에 처음 연결된 장치는 아직 자신의 IP 주소를 알지 못한다.
    - 따라서 특정 서버를 지정해 요청할 수 없기 때문에 네트워크 전체에 요청을 보내야 한다.
    - 이때 사용하는 방식이 Broadcast이다.
    - DHCP 동작 흐름 (간략)
      - DHCP Discover
        - 클라이언트가 네트워크 전체에 Broadcast로 IP 주소를 요청한다.
      - DHCP Offer
        - DHCP 서버가 사용 가능한 IP 주소 정보를 응답한다.
      - DHCP Request
        - 클라이언트가 해당 IP 주소를 사용하겠다고 요청한다.
      - DHCP Ack
        - DHCP 서버가 최종적으로 IP 주소와 네트워크 설정을 할당한다.
  - 이 과정을 통해 클라이언트는 자동으로 네트워크 설정을 받아 인터넷을 사용할 수 있게 된다.

### ARP
- ARP (Address Resolution Protocol)
  - IP 주소를 이용하여 해당 장치의 MAC 주소를 알아내기 위한 프로토콜이다.
- MAC 주소를 알아내야 하는 이유
  - 네트워크 통신은 계층적으로 이루어진다.
  - IP 주소는 L3(Network Layer)에서 사용하는 논리 주소이다.
  - 실제 네트워크에서 데이터를 전달하는 L2(Data Link Layer)에서는 MAC 주소를 사용한다.
  - 따라서 IP Packet을 실제 네트워크로 전송하려면 목적지 장비의 MAC 주소가 필요하다.
- 동작 예시
  - 구성
    - 내 PC
    - Gateway(라우터)
    - 네이버 서버 (IP: 3.3.3.3 라고 가정)
  - 네이버 서버에 접속할 때의 흐름
    - 사용자가 네이버에 접속하면 목적지 IP는 `3.3.3.3`이 된다.
    - 하지만 네이버 서버는 다른 네트워크에 있기 때문에 직접 통신할 수 없다.
    - 따라서 패킷은 먼저 Gateway(라우터)로 전달되어야 한다.
  - 이때 패킷의 주소 구조는 다음과 같이 구성된다.
    - IP Header (L3)
      - Source IP : 내 PC
      - Destination IP : 3.3.3.3 (네이버 서버)
    - Frame Header (L2)
      - Source MAC : 내 PC
      - Destination MAC : Gateway
  - 즉
    - IP 주소 기준 목적지는 네이버 서버이지만
    - 실제 네트워크 전송은 Gateway를 통해 이루어진다.

### Ping과 RTT
- Ping
  - 네트워크에서 특정 Host와의 연결 상태를 확인하기 위해 사용하는 명령어이다.
  - 대상 Host까지 패킷이 정상적으로 전달되는지 확인하고, 응답 시간을 측정할 수 있다.
  - Ping은 ICMP(Internet Control Message Protocol)를 이용하여 동작한다.
- 동작 방식
  - Ping을 실행하면 대상 Host로 ICMP Echo Request 패킷을 전송한다.
  - 대상 Host는 이를 수신하면 ICMP Echo Reply로 응답한다.
  - 이 요청과 응답 과정을 통해 통신 가능 여부와 응답 시간을 확인할 수 있다.
- RTT (Round Trip Time)
  - 패킷이 송신자에서 목적지로 전달되고 다시 돌아오기까지 걸린 시간을 의미한다.
  - Ping은 이 RTT 값을 측정하여 네트워크 지연 시간을 확인하는 데 사용된다.

### TCP와 UDP 개요
- TCP (Transmission Control Protocol)
  - 연결 지향(Connection-oriented) 프로토콜이다.
  - 데이터를 전송하기 전에 송신자와 수신자 사이에 연결을 먼저 설정한다.
  - 데이터의 순서를 보장하며, 유실된 데이터는 재전송한다.
  - 수신 확인(ACK), 흐름 제어, 혼잡 제어 등의 기능을 제공한다.
  - 신뢰성이 중요한 통신에서 사용된다.
  - 예: 웹(HTTP/HTTPS), 파일 전송, 이메일 등
- UDP (User Datagram Protocol)
  - 비연결형(Connectionless) 프로토콜이다.
  - 연결 설정 없이 데이터를 바로 전송한다.
  - 데이터의 순서 보장이나 재전송 기능이 없다.
  - 구조가 단순하여 속도가 빠르고 오버헤드가 적다.
  - 실시간성이 중요한 통신에서 사용된다.
  - 예: DNS, 스트리밍, 실시간 게임, VoIP 등

### TCP 연결과정
- 3-way Handshaking
  - TCP에서 송신자와 수신자가 데이터를 주고받기 전에 연결을 설정하는 과정이다.
  - 양쪽이 서로 통신 가능한 상태인지 확인하고 초기 시퀀스 번호를 교환한다.
- 동작 과정
  - 1. SYN
    - 클라이언트가 서버에게 연결을 요청한다.
    - 이때 SYN(Synchronize) 패킷을 전송한다.
    - 패킷에는 클라이언트의 초기 Sequence Number가 포함된다.
  - 2. SYN + ACK
    - 서버는 연결 요청을 수신하면 응답을 보낸다.
    - SYN과 ACK가 함께 설정된 패킷을 전송한다.
    - ACK는 클라이언트의 SYN을 정상적으로 받았다는 의미이다.
    - 동시에 서버의 초기 Sequence Number도 전달된다.
  - 3. ACK
    - 클라이언트가 서버의 SYN을 확인하고 ACK 패킷을 전송한다.
    - 이 과정이 완료되면 TCP 연결이 성립된다.
- 정리
  - Client → Server : SYN
  - Server → Client : SYN + ACK
  - Client → Server : ACK
- 이때 전송되는 세그먼트의 구조
  - 3-way Handshaking에서 사용되는 세그먼트는 연결 설정을 위한 제어 패킷이다.
  - 따라서 일반적인 데이터 전송과 달리 Payload는 포함되지 않는다.
  - 보통 다음과 같은 구조로 전송된다.
    - IP Header
    - TCP Header
    - Payload 없음
  - 참고
    - 이론적으로는 SYN 패킷에 데이터가 포함될 수도 있다.
    - 하지만 일반적인 TCP 연결 과정에서는 Payload 없이 사용되는 것이 일반적이다.

### TCP 연결종료 및 상태변화
- 4-way Handshaking
  - TCP 연결을 정상적으로 종료하기 위해 사용하는 과정이다.
  - 연결은 양방향 통신이기 때문에 한쪽에서 종료 요청을 하더라도 반대 방향의 통신이 남아 있을 수 있다.
  - 따라서 송신 방향과 수신 방향을 각각 종료하는 절차가 필요하며, 이 때문에 4단계 과정으로 진행된다.
- 동작 과정
  - 1. FIN
    - 클라이언트가 연결 종료를 요청한다.
    - TCP Header에 FIN 플래그가 설정된 세그먼트를 전송한다.
    - 이 시점부터 클라이언트는 더 이상 데이터를 보내지 않는다.
  - 2. ACK
    - 서버는 클라이언트의 FIN을 수신하고 ACK로 응답한다.
    - 이는 종료 요청을 정상적으로 받았다는 의미이다.
    - 하지만 서버는 아직 데이터를 전송할 수 있는 상태이다.
  - 3. FIN
    - 서버도 데이터 전송이 끝나면 연결 종료를 요청한다.
    - TCP Header에 FIN 플래그를 설정하여 클라이언트에게 전송한다.
  - 4. ACK
    - 클라이언트는 서버의 FIN을 수신하고 ACK로 응답한다.
    - 이 과정이 완료되면 TCP 연결이 종료된다.
- 정리
  - Client → Server : FIN
  - Server → Client : ACK
  - Server → Client : FIN
  - Client → Server : ACK
- 상태 변화 예시 (간략)
  - Client
    - ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
  - Server
    - ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
- TIME_WAIT 상태
  - 마지막 ACK 이후 일정 시간 동안 연결 정보를 유지하는 상태이다.
  - 네트워크 지연으로 인해 남아 있을 수 있는 패킷 문제를 방지하기 위한 상태이다.

### TCP, UDP 헤더형식과 게임서버 특징
- TCP Header 형식
  - TCP는 신뢰성 있는 전송을 위해 다양한 제어 정보를 포함한다.
  - 기본 헤더 크기는 20byte이며, 옵션이 포함되면 더 커질 수 있다.
  - 주요 필드
    - Source Port
      - 송신 측 포트 번호
    - Destination Port
      - 수신 측 포트 번호
    - Sequence Number
      - 전송되는 데이터의 순서를 나타내는 번호
    - Acknowledgment Number
      - 정상적으로 수신한 데이터의 다음 번호
    - Header Length (Data Offset)
      - TCP 헤더의 길이
    - Flags
      - 제어 비트 (SYN, ACK, FIN, RST, PSH, URG 등)
    - Window Size
      - 수신 측이 처리 가능한 데이터 크기 (흐름 제어)
    - Checksum
      - 데이터 오류 검증
    - Urgent Pointer
      - 긴급 데이터 위치
    - Options
      - 추가 기능 설정 (MSS, Window Scale 등)
- UDP Header 형식
  - UDP는 구조가 단순하며 빠른 전송을 목적으로 한다.
  - 헤더 크기는 고정 8byte이다.
  - 주요 필드
    - Source Port
      - 송신 측 포트 번호
    - Destination Port
      - 수신 측 포트 번호
    - Length
      - UDP 헤더 + 데이터의 전체 길이
    - Checksum
      - 데이터 오류 검증
- TCP와 UDP 헤더 차이
  - TCP
    - 헤더 구조가 복잡하다.
    - 순서 제어, 재전송, 흐름 제어 기능이 포함된다.
  - UDP
    - 헤더 구조가 단순하다.
    - 최소한의 정보만 포함하고 빠르게 전송한다.

### TCP '연결'이라는 착각
- TCP 연결이라는 개념
  - TCP는 흔히 "연결 기반 프로토콜"이라고 불린다.
  - 하지만 실제로 물리적인 연결이 존재하는 것은 아니다.
  - TCP는 단지 양쪽 Host가 서로의 상태 정보를 유지하며 통신하는 논리적인 연결이다.
- 재전송 타이머 (Retransmission Timer)
  - TCP는 데이터가 정상적으로 전달되었는지 확인하기 위해 ACK 응답을 기다린다.
  - 패킷을 전송하면 일정 시간 동안 ACK를 기다리는 타이머가 동작한다.
  - 이 타이머를 재전송 타이머라고 한다.
- RTO (Retransmission Timeout)
  - 재전송 타이머가 기다리는 시간을 RTO라고 한다.
  - 일정 시간(RTO) 동안 ACK를 받지 못하면 패킷이 유실된 것으로 판단한다.
  - 이 경우 동일한 데이터를 다시 전송한다.
- 재전송 동작
  - 송신 측은 패킷을 전송하고 ACK를 기다린다.
  - RTO 시간 안에 ACK를 받지 못하면 재전송을 수행한다.
  - 이러한 재전송은 일반적으로 최대 5회 정도 시도된다.
  - 모든 재전송이 실패하면 연결 문제로 판단하고 전송 오류가 발생한다.

### 한 번에 끝내는 DNS
- DNS (Domain Name System)
  - 사람이 이해하기 쉬운 도메인 이름을 IP 주소로 변환해주는 시스템이다.
  - 예를 들어 `www.naver.com`을 입력하면 실제 통신에 필요한 IP 주소를 찾아준다.
- 분산 구조형 데이터베이스
  - DNS는 하나의 서버가 모든 정보를 관리하지 않는다.
  - 전 세계에 분산된 여러 DNS 서버가 함께 동작하는 구조이다.
  - 데이터베이스 시스템이 여러 서버에 나뉘어 분산 구성된다.
  - 도메인 영역별로 관리 서버가 나뉘어 있으며 각각 독립적으로 관리된다.
  - 이를 통해 트래픽을 분산하고 안정성을 높인다.
- 트리 구조의 도메인 네임 체계
  - 트리 구조
    - DNS는 계층적인 트리 구조로 이루어진다.
      - 예시 : www.naver.com
        - `.` → Root Domain
        - `com` → Top Level Domain (TLD)
        - `naver` → Domain Name
        - `www` → Host Name
    - 일반적으로
      - `naver.com` → 도메인 네임
      - `www` → 호스트 네임
  - 도메인 네임 체계
    - 모든 도메인 이름은 전 세계에서 유일해야 한다.
    - DNS는 계층적인 구조를 통해 도메인 이름이 중복되지 않도록 관리된다.
- DNS 조회 흐름
  - 사용자가 `www.naver.com`을 입력하면 다음과 같은 과정이 발생한다.
  - 1. 로컬 DNS 확인
    - 먼저 PC 내부의 DNS 캐시를 확인한다.
    - 이미 조회된 기록이 있으면 바로 IP 주소를 반환한다.
  - 2. ISP DNS 서버 질의
    - 캐시에 정보가 없다면 ISP(Internet Service Provider)의 DNS 서버에 질의한다.
    - ISP DNS는 보통 통신사나 네트워크 제공자가 운영하는 DNS 서버이다.
  - 3. Root DNS 서버 조회
    - ISP DNS 서버도 모르면 Root DNS 서버에 질의한다.
    - Root 서버는 해당 TLD 서버 위치를 알려준다.
  - 4. TLD DNS 서버 조회
    - Root 서버로부터 `com` TLD 서버 정보를 받는다.
    - 이후 `com` DNS 서버에 `naver.com`에 대한 정보를 질의한다.
  - 5. Authoritative DNS 서버 조회
    - `naver.com`을 관리하는 권한 서버(Authoritative DNS)를 찾는다.
    - 이 서버가 `www.naver.com`의 실제 IP 주소를 반환한다.
  - 6. 응답 반환
    - DNS 서버는 조회된 IP 주소를 사용자에게 전달한다.
    - 이후 클라이언트는 해당 IP로 실제 통신을 시작한다.
- DNS 캐시와 유효시간
  - DNS 조회 결과는 캐시에 저장된다.
  - 동일한 도메인을 다시 조회할 때 빠르게 응답하기 위함이다.
  - DNS 응답에는 TTL(Time To Live) 값이 포함된다.
    - 이 값은 캐시가 유효한 시간을 의미한다.
    - TTL 시간이 지나면 캐시가 만료되어 다시 DNS 조회가 수행된다.

### URL과 URI
- 정의
  - URI (Uniform Resource Identifier)
    - 인터넷에 존재하는 자원을 식별하기 위한 문자열 규칙이다.
    - 어떤 자원이 어디에 있는지 또는 어떤 방식으로 접근할 수 있는지를 표현한다.
    - 웹 주소를 표현하는 상위 개념이다.
  - URL (Uniform Resource Locator)
    - 자원의 위치(Location)를 나타내는 URI의 한 종류이다.
    - 네트워크 상에서 자원이 어디에 존재하는지와 접근 방법을 함께 나타낸다.
- 포괄 관계
  - URI가 상위 개념이다.
  - URL은 URI에 포함되는 개념이다.
  - 즉 모든 URL은 URI이지만, 모든 URI가 URL인 것은 아니다.
  - 예시
    - `https://www.naver.com`
      - 자원의 위치를 나타내므로 URL이며 동시에 URI이다.
- URI 구조
  - URI는 일반적으로 다음과 같은 구조로 구성된다.
  - `scheme ":" "//" authority "/" path "?" query "#" fragment`
  - 주요 구성 요소
    - scheme
      - 자원에 접근하기 위한 프로토콜을 의미한다.
      - 예: http, https, ftp 등
    - authority
      - 서버의 주소를 의미한다.
      - 보통 도메인 이름이나 IP 주소가 들어간다.
    - path
      - 서버 내부에서 자원의 위치를 나타낸다.
    - query
      - 서버에 전달되는 추가 파라미터이다.
      - 보통 `?key=value` 형태로 전달된다.
    - fragment
      - 문서 내부의 특정 위치를 가리킨다.
      - `#` 뒤에 오는 값이다.

### 굵고 짧게 살펴보는 HTTP

- HTTP 개요
  - HTTP(HyperText Transfer Protocol)는 웹에서 클라이언트와 서버가 데이터를 주고받기 위해 사용하는 애플리케이션 계층 프로토콜이다.
  - 주로 브라우저와 웹 서버 사이의 통신에 사용된다.
  - 요청(Request)과 응답(Response) 구조로 동작한다.
  - 기본적으로 Stateless 특성을 가지며, 각 요청은 독립적으로 처리된다.
- 헤더
  - HTTP 메시지는 크게 Start Line, Header, Body로 구성된다.
  - Header에는 요청이나 응답 처리에 필요한 메타 정보가 포함된다.
  - 종류
    - 일반 헤더 (General Header)
      - 요청과 응답 모두에서 공통적으로 사용되는 헤더이다.
      - 메시지 전체와 관련된 정보를 전달한다.
      - 예: Date, Connection 등
    - 요청 헤더 (Request Header)
      - 클라이언트가 서버로 요청을 보낼 때 포함되는 헤더이다.
      - 클라이언트 환경이나 요청 조건에 대한 정보를 전달한다.
      - 예: Host, User-Agent, Accept 등
    - 응답 헤더 (Response Header)
      - 서버가 클라이언트에게 응답할 때 포함되는 헤더이다.
      - 서버 정보나 응답 처리와 관련된 정보를 전달한다.
      - 예: Server, Set-Cookie, Location 등
    - 엔티티 헤더 (Entity Header)
      - 메시지 Body에 포함된 데이터에 대한 정보를 설명한다.
      - 콘텐츠 타입이나 길이 등을 정의한다.
      - 예: Content-Type, Content-Length 등
- 메서드
  - HTTP 메서드는 클라이언트가 서버에게 어떤 동작을 요청하는지 나타낸다.
    - GET
      - 서버로부터 데이터를 조회할 때 사용한다.
    - POST
      - 서버에 데이터를 전송하여 새로운 자원을 생성할 때 사용한다.
    - PUT
      - 특정 자원을 생성하거나 전체 수정할 때 사용한다.
    - PATCH
      - 자원의 일부를 수정할 때 사용한다.
    - DELETE
      - 특정 자원을 삭제할 때 사용한다.
- 응답 코드
  - 200번대 (성공)
    - 200 OK
      - 요청이 정상적으로 처리되었음을 의미한다.
    - 201 Created
      - 요청이 성공하여 새로운 자원이 생성되었음을 의미한다.
    - 204 No Content
      - 요청은 성공했지만 반환할 데이터가 없음을 의미한다.
  - 300번대 (리다이렉션)
    - 301 Moved Permanently
      - 요청한 자원이 영구적으로 다른 URL로 이동했음을 의미한다.
    - 302 Found
      - 요청한 자원이 일시적으로 다른 위치로 이동했음을 의미한다.
    - 304 Not Modified
      - 캐시된 자원을 그대로 사용해도 된다는 의미이다.
  - 400번대 (클라이언트 오류)
    - 400 Bad Request
      - 요청 형식이 잘못된 경우 발생한다.
    - 401 Unauthorized
      - 인증이 필요하지만 인증 정보가 없는 경우 발생한다.
    - 403 Forbidden
      - 요청은 이해했지만 접근 권한이 없는 경우 발생한다.
    - 404 Not Found
      - 요청한 자원을 찾을 수 없는 경우 발생한다.
  - 500번대 (서버 오류)
    - 500 Internal Server Error
      - 서버 내부 오류로 요청을 처리할 수 없는 경우 발생한다.
    - 502 Bad Gateway
      - 게이트웨이 또는 프록시 서버가 잘못된 응답을 받은 경우 발생한다.
    - 503 Service Unavailable
      - 서버가 과부하 상태이거나 일시적으로 요청을 처리할 수 없는 경우 발생한다.
    - 504 Gateway Timeout
      - 게이트웨이 서버가 응답을 제시간에 받지 못한 경우 발생한다.
